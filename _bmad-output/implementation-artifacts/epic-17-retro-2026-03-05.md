## Défis rencontrés

| Défi | Story | Solution |
|------|-------|---------|
| Adaptation TerminalGridView (reveal+guessing) | 17.2, 17.3 | Reset via `.id()` — non-évident |
| Offset validation off-by-one | 17.3 | `poolOffset = revealedCount + guessingInput.count` |
| UserDefaults piège score=0 | 17.4 | `.object(forKey:) != nil` avant `.integer(forKey:)` |
| Deep link cold start timing | 17.5 | Delegate dans `App.init()`, pas view lifecycle |
| Haptic feedback doublé | 17.3 | Split playSuccess/playDoubleBang |
| fullScreenCover + NavigationCoordinator | 17.4 | Close overlay → 100ms delay → pop |

**Thème récurrent :** 3/6 stories ont eu des issues liées au timing/état SwiftUI.

---

## Insights clés

1. Le pattern `.id()` pour reset d'état SwiftUI interne devrait être documenté dans le project-context
2. Le code review est le dernier rempart qualité — ne jamais raccourcir le process
3. Les tests déterministes sont supérieurs aux tests random pour la logique mathématique (MUS)
4. L'isolation des guards dans les tests de state machine prévient les faux positifs

## Patterns réutilisables documentés

| Pattern | Description | Usage |
|---------|-------------|-------|
| Backward Compat Codable | `decodeIfPresent` + defaults | Extension de modèles persistés |
| Singleton avec injection | `init(defaults: .standard)` | Testabilité UserDefaults |
| Deep Link Enum | `@Published var pendingDeepLink: DeepLink?` | Navigation type-safe |
| State Machine Guards | Guards explicites entre phases | Prévention transitions invalides |
| `.id()` Reset | `.id(stateKey)` sur composants SwiftUI | Reset état interne drift |
| Score Bonus/Malus | `correct * 10 - hints * 5` | Scores négatifs = signal |
